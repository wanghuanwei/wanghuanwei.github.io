ReetrantLock浅析

发表于 2019-05-08 | 分类于 JUC

写在前面
一直想写blog，但一直很懒。直到真的面临中年危机才发现很多时候，自己的沉淀真的很薄弱，所以下定决心开始沉淀自己。本来第一篇想写Spring相关的，但确实是内容太多。正好今天又莫名的看到ReentrantLock，写这个也好像不错。

正文：
在Java中，悲观锁通常的实现方式有两种：synchronized和ReentrantLock。而相对来说，Lock的方式更方便一些，可以指定一些参数，比如是否是公平锁（构造参数指定），等待时长等，所以先简单写一下ReentrantLock。
说到ReetrantLock，就必须要说AQS（AbstractQueuedSynchronizer），它是ReentrantLock的实现基础。AQS通过一个声明为volatile的state和一个CLH队列来管理锁定的对象，此处就不展开写了。
在new一个ReentrantLock时，可以指定锁是公平锁还是非公平锁。如下所示：


    public ReentrantLock() {
        sync = new NonfairSync();
    }
    public ReentrantLock(boolean fair) {
    // 通过构造方法的入参来指定是否是公平锁，默认是NonFair
        sync = fair ? new FairSync() : new NonfairSync();
    }

默认的就是一个非公平锁。所谓公平和非公平，就是看等待的线程在竞争锁时的动作。公平锁就是大家都待在队列中，依次来获取锁资源；非公平锁则在请求锁时，先来尝试获取依次锁，如果能不能获取到再来加入等待队列，而不是按照队列的顺序依次获取锁资源。当然这个是概念上的区别，但代码又是如何的不同呢，所以往下看。（不容易，铺垫这么多，终于到我想记录的了。。看来自己还是真的啰嗦）
先讲公共的部分
在使用锁时，如果是调用tryLock()方法时，其实都是调用的非公平的获取锁的方式，最终代码如下：

  final boolean nonfairTryAcquire(int acquires) {
            final Thread current = Thread.currentThread();
            int c = getState();
            if (c == 0) {
            // 获取当前锁状态，如果为0，表示暂没人抢占，直接通过CAS来抢占锁，成功则返回true
                if (compareAndSetState(0, acquires)) {
                    setExclusiveOwnerThread(current);
                    return true;
                }
            }
            else if (current == getExclusiveOwnerThread()) {
            // 如果锁被抢占则判断是否是当前线程，如果是，则将state加上要申请的数量-acquires，并返回true
                int nextc = c + acquires;
                if (nextc < 0)
                    throw new Error(“Maximum lock count exceeded”);
                setState(nextc);
                return true;
            }
            // 否则返回失败
            return false;
        }

其实上面的代码是看不出来公平不公平的，任何的线程来申请锁时，肯定都是这种步骤：如果没有被抢占，则尝试抢占，如果被占则判断是否是自身。这个也是可重入锁的根本原因。
在真正调用tryLock(long timeout, TimeUnit unit) 这个方法时，公平和非公平才会体现出来。
先来看默认的非公平锁。
无论是Fair还是NonFair，在tryLock（timeout，unit）时，都是调用的AQS的tryAcquireNanos(arg，timeout)这个方法，代码如下：

   public final boolean tryAcquireNanos(int arg, long nanosTimeout)
            throws InterruptedException {
        if (Thread.interrupted())
            throw new InterruptedException();
            // 此处的tryAcquire才会区分Fair和NonFair，其中arg默认为1
        return tryAcquire(arg) ||
            doAcquireNanos(arg, nanosTimeout);
    }

NonFair的tryAcquire实现，就是上面tryLock()的代码，在锁正好处于未锁定时，直接获取锁。
再来看Fair的实现：

   protected final boolean tryAcquire(int acquires) {
            final Thread current = Thread.currentThread();
            int c = getState();
            if (c == 0) {
            // 公平锁是在看到锁未被占用时，先判断是否有等待队列且队列非空，如果是才尝试获取锁，如果不是，则返回false进入后续处理流程
                if (!hasQueuedPredecessors() &&
                    compareAndSetState(0, acquires)) {
                    setExclusiveOwnerThread(current);
                    return true;
                }
            }
            else if (current == getExclusiveOwnerThread()) {
            // 以下和NonFair的处理一致
                int nextc = c + acquires;
                if (nextc < 0)
                    throw new Error(“Maximum lock count exceeded”);
                setState(nextc);
                return true;
            }
            return false;
        }

可以看到Fair和NonFair的差别主要就是体现在tryAcquire方法中，如果锁为被抢占时的处理方式不一样，Fair会主动入队，而NonFair则是直接抢占。这个也会影响入队后的处理。
